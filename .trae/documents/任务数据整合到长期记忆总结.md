# 任务数据整合到长期记忆总结 - 实施计划

## 一、需求概述

在执行长期记忆总结功能时，同步对任务数据进行总结处理，确保每日产生的任务数据能够完整、准确地被整合并存储到长期记忆系统中。

## 二、当前问题分析

### 2.1 现有数据流

```
短期记忆(messages + practiceSummaries) 
    → generateSummary() 
    → DailySummary 
    → 长期记忆
```

### 2.2 缺失环节

- **任务数据未参与总结**：`ai_study_tasks` 中的任务记录未被读取
- **`tasksCompleted` 字段来源不准确**：由 LLM 从对话推断，非真实任务数据
- **启动检测服务同样缺失任务处理**

## 三、修改范围

### 3.1 需要修改的文件

| 文件路径 | 修改内容 |
|---------|---------|
| `apps/web/src/services/memory-archive.ts` | 核心归档服务，添加任务数据读取和整合 |
| `apps/web/src/services/memory-startup-check.ts` | 启动检测服务，同步添加任务处理 |

### 3.2 不需要修改的文件

| 文件路径 | 原因 |

#### 1.2 新增任务数据格式化函数

添加新函数 `buildTaskDataPrompt`，用于将任务数据格式化为 LLM 可理解的文本：

```typescript
function buildTaskDataPrompt(tasks: Task[]): string {
  if (!tasks || tasks.length === 0) {
    return ''
  }
  
  let text = '\n\n【当日任务记录】\n'
  
  const completedTasks = tasks.filter(t => t.status === '已完成')
  const pendingTasks = tasks.filter(t => t.status === '未完成')
  
  if (completedTasks.length > 0) {
    text += `\n已完成任务 (${completedTasks.length}个):\n`
    completedTasks.forEach((task, index) => {
      text += `${index + 1}. ${task.name}`
      text += ` [${task.type}]`
      if (task.estimatedTime) {
        text += ` 预计${task.estimatedTime}分钟`
      }
      if (task.completedAt) {
        text += ` 完成于${task.completedAt.split('T')[1]?.slice(0, 5) || ''}`
      }
      text += '\n'
    })
  }
  
  if (pendingTasks.length > 0) {
    text += `\n未完成任务 (${pendingTasks.length}个):\n`
    pendingTasks.forEach((task, index) => {
      text += `${index + 1}. ${task.name}`
      text += ` [${task.type}]`
      if (task.estimatedTime) {
        text += ` 预计${task.estimatedTime}分钟`
      }
      text += '\n'
    })
  }
  
  return text
}
```

## 五、数据流变更对比

### 修改前

```
短期记忆(messages + practiceSummaries) 
    → generateSummary() 
    → DailySummary(tasksCompleted: LLM推断)
    → 长期记忆
```

### 修改后

```
短期记忆(messages + practiceSummaries) + 任务数据(tasks)
    → generateSummary() 
    → DailySummary(tasksCompleted: 基于真实任务数据)
    → 长期记忆
```

## 六、边界与风险评估

### 6.1 功能边界

| 边界项 | 说明 |
|-------|------|
| 只读操作 | 任务数据仅用于总结，不会被修改或删除 |
| 向后兼容 | 如果任务数据为空，总结流程正常执行 |
| 模块隔离 | 修改仅限于记忆归档服务，不影响任务管理模块 |

### 6.2 风险评估

| 风险 | 级别 | 缓解措施 |
|-----|------|---------|
| 任务数据读取失败 | 低 | 使用 try-catch 包裹，失败时返回空数组 |
| LLM 仍可能忽略任务数据 | 中 | 在系统提示词中明确强调任务数据来源 |
| 增量合并时任务重复 | 低 | 使用 Set 去重 |

## 七、测试验证要点

1. **单元测试**：验证 `buildTaskDataPrompt` 函数输出格式正确
2. **集成测试**：验证完整归档流程中任务数据被正确读取和整合
3. **边界测试**：
   - 无任务数据时的归档流程
   - 只有已完成任务的情况
   - 只有未完成任务的情况
4. **回归测试**：确保现有对话总结和练习总结功能不受影响
